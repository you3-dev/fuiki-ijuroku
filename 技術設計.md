# 『封域異獣録』技術設計

版：0.2  
更新日：2026年7月13日  
対象：縦切り版（iPhone向けPWA／Windowsで開発）

## 1. 採用構成

| 領域 | 採用技術 | 主な役割 |
|---|---|---|
| UI | React＋TypeScript | 研究所、地点選択、戦闘、異獣録、共鳴継承の画面 |
| 開発基盤 | Vite | 開発サーバー、ビルド、静的素材の処理 |
| ゲームルール | フレームワーク非依存のTypeScript | 戦闘、探索、仲間化、育成、共鳴継承の計算と状態遷移 |
| 保存 | Dexie（IndexedDB） | セーブ、設定、図鑑、調査進行、バックアップ用データ |
| PWA | vite-plugin-pwa | Manifest、Service Worker、オフライン起動、更新制御 |
| 大画面の遷移 | React Router（Declarative Mode／HashRouter） | 戻る操作、内部リンク、直接起動時の安全な補正 |
| 単体テスト | Vitest | 計算式、状態遷移、セーブ移行の自動テスト |
| 結合・画面テスト | Playwright | 主要な一周動作、WebKit、スマートフォン相当表示の確認 |
| 実機確認 | iPhoneのSafari／ホーム画面追加版 | 最終的な操作性、音声、保存、中断復帰、オフライン動作の判定 |

縦切り版ではPhaserを導入しない。Reactの通常UIで地点選択方式を作り、プレイテスト後にタイルマップ自由移動が必要と判断した場合だけ、探索表示の実装候補として追加する。

バージョン番号は設計書へ固定せず、実装開始時点の安定版を採用してロックファイルへ記録する。開発用Node.jsは、その時点でViteが公式に要求するバージョンを使用する。

## 2. 設計方針

### 2.1 ゲームルールをReactから分離する

戦闘計算や進行判定は、Reactコンポーネント内へ直接書かない。ゲームルール層はReact、Dexie、ブラウザーAPIを参照しない純粋なTypeScriptとする。

これにより次が可能になる。

- 画面を作る前に戦闘や継承を自動テストする
- 地点選択画面をPhaserのタイルマップへ変更しても、地域データと進行条件を再利用する
- セーブデータの読み書きと、ゲーム上の計算を別々に修正する
- 乱数シードを指定し、戦闘不具合を再現する

### 2.2 状態変更はコマンドとイベントで表す

画面からゲーム状態を直接書き換えず、次の流れに統一する。

```text
ユーザー操作
  → コマンド生成
  → ゲームルール層が妥当性を検証
  → 新しい状態と発生イベントを返す
  → 確定単位で保存
  → Reactが結果を表示
```

例：戦闘の「ラウンド確定」は、3体分の行動を含む1コマンドとして処理する。計算途中の状態をセーブせず、ラウンド開始前または解決後の一貫した状態だけを保存する。

### 2.3 グローバルな可変状態を作らない

ゲーム全体を書き換え可能な単一オブジェクトとして公開しない。状態変更はルール層の関数またはセッション管理の入口に限定する。TypeScriptは厳格設定を使用し、不明なデータを型検証なしでセーブへ入れない。

## 3. 推奨ディレクトリ

```text
src/
  app/                    # 起動処理、画面遷移、全体レイアウト
  domain/                 # ReactやDBに依存しないゲームルール
    battle/
    exploration/
    recruitment/
    breeding/
    growth/
    bestiary/
  data/                   # 異獣、技能、地域、イベントの定義データ
  infrastructure/         # ブラウザー機能との接続
    db/
    pwa/
    audio/
    backup/
  features/               # プレイヤーが操作する画面単位
    laboratory/
    battle/
    exploration/
    bestiary/
    resonance/
    settings/
  assets/                 # 立ち絵、アイコン、背景、音声
  test/                   # 複数機能をまたぐテスト用データ
```

`domain` から `features` や `infrastructure` を参照しない。画面や保存側から `domain` を呼び出す一方向の依存にする。

## 4. 探索方式の交換境界

地域データは表示方式から独立させ、最低限次を持つ。

- 地点ID、名称、説明、背景ID
- 地点間の接続
- 進入条件
- 調査選択肢と発生条件
- 遭遇候補
- 取得記録、素材、進行フラグ
- 調査済み状態

探索画面は、共通の探索操作をルール層へ渡す役割に限定する。

```ts
type ExplorationCommand =
  | { type: 'enterNode'; nodeId: string }
  | { type: 'chooseAction'; actionId: string }
  | { type: 'finishEncounter'; encounterId: string };
```

縦切り版ではReact製の `NodeExplorationView` が地点一覧と短い分岐を表示する。タイルマップ化する場合は、地域データからマップ上の入口やイベントを対応付ける `TileExplorationView` を追加する。進行フラグ、遭遇、報酬、物語イベントの判定は共通のままにする。

Phaserを追加するとしても、研究所、戦闘コマンド、異獣録、共鳴継承までCanvasへ移す必要はない。自由移動が必要な探索部分だけを担当させる。

## 5. セーブ設計

### 5.1 保存先

Dexieを通してIndexedDBへ保存する。Dexieのテーブル構成とゲーム内セーブ形式は分け、内部DBを変更してもJSONバックアップ形式を移行できるようにする。

想定テーブルは次のとおり。

| テーブル | 内容 |
|---|---|
| `profiles` | 主人公設定、物語進行、編成、研究資源 |
| `creatures` | 所有個体、能力、技能、信頼度、系譜 |
| `bestiary` | 種ごとの調査段階、観察済み情報 |
| `expeditions` | 地域進行、地点状態、確定済みの調査状況 |
| `settings` | 音量、表示、操作設定 |
| `metadata` | スキーマ版、アプリ版、最終保存日時 |

縦切り版は1プロフィールを前提とするが、各データには安定したIDを付ける。

### 5.2 保存の確定単位

- 主人公設定の完了
- 編成または育成の確定
- 調査開始と地点イベントの完了
- 戦闘ラウンド解決後
- 戦闘終了後
- 仲間化完了後
- 共鳴継承完了後
- 設定変更後

複数テーブルを更新する操作は、Dexieのトランザクションで一括確定する。処理失敗時に異獣だけ増え、素材だけ減るような部分保存を防ぐ。

### 5.3 バージョン移行とバックアップ

- 全セーブに `schemaVersion` を持たせる
- DB更新ごとに旧版からの移行処理を用意する
- JSON読込時は形式、版、必須値、参照IDを検証する
- 復元前に現在セーブを書き出せるよう案内する
- 読込に失敗したデータで現在セーブを上書きしない
- 将来版のセーブは、破壊的に読み込まず対応外と表示する

## 6. PWAと更新

Manifestの基本値は次とする。

| 項目 | 値 |
|---|---|
| `name` | `封域異獣録` |
| `short_name` | `異獣録` |
| `display` | `standalone` |
| 基準画面 | 縦画面 |
| 起動先 | アプリのホーム画面 |

アプリ本体、UIアイコン、縦切り版に必須の画像・音声をオフライン利用可能にする。ただし、不要な制作素材や将来地域の素材は事前キャッシュしない。

新しいService Workerを検出しても、戦闘中、調査イベント処理中、共鳴継承の確認中には強制反映しない。安全な画面で「更新して再起動」を選べるようにする。更新前には現在の操作を確定保存する。

## 7. テスト方針

### Vitestで優先するもの

- ダメージ、回復、状態持続時間の境界値
- 活性値の増減と上限・下限
- 3体分の行動順と戦闘不能時の処理
- 観察条件、鎮静、協力要請の成立判定
- 共鳴継承の容量計算と提供元非消失
- 成長曲線、経験値配分
- 灰苔湿原の進入条件とボス3段階
- 同一シードで同一結果になること
- セーブ形式の版移行

### Playwrightで優先する一周動作

1. 主人公設定を完了する
2. 編成して灰苔湿原へ出る
3. 調査、戦闘、観察、仲間化を行う
4. アプリを再読込して確定地点から再開する
5. 先遣隊の記録を集めてニゴリグイを鎮静する
6. 共鳴継承を行い、提供元2体が残ることを確認する
7. JSONを書き出し、別の空セーブへ復元する

ChromiumとWebKitで自動確認する。PlaywrightのWebKit確認はSafari相当の早期発見に使うが、iPhone上のSafariそのものを保証するものとはみなさない。

### iPhone実機で必ず確認するもの

- ホーム画面追加と独立起動
- セーフエリア、文字サイズ、タップ領域
- 画面ロック、アプリ切替、再開
- 機内モードでの再起動
- 音声開始条件、消音、バックグラウンド復帰
- 長時間遊んだ後の保存容量と読み込み
- アプリ更新後の旧セーブ移行

## 11. 次に決めること

アプリ起動中の状態はReact標準の `useReducer`＋`Context` で保持し、確定単位で `SaveCoordinator` からDexieへ直列保存する。詳細は [状態管理設計.md](./状態管理設計.md) を参照する。

大画面だけをReact Routerの `HashRouter` でURLへ対応させ、細かなコマンド入力はURL化しない。詳細は [画面遷移設計.md](./画面遷移設計.md) を参照する。

通常画面は研究所、編成、異獣録の下部3タブ、探索と戦闘はタブを隠した集中画面とする。詳細は [画面構成設計.md](./画面構成設計.md) を参照する。

UIの視覚表現は [ビジュアルデザイン.md](./ビジュアルデザイン.md)、画面の具体化は [画面構成設計.md](./画面構成設計.md) と [研究所ホーム設計.md](./研究所ホーム設計.md) を参照する。PWA保存基盤、固定チュートリアル戦、観測櫓跡の汎用戦闘、沈み水路の複数対象・汚染戦、濾過樹群の固定条件接触と中枢解放は実装済みである。次の技術工程は、既存の2段階ラウンド保存を使ってニゴリグイの3段階ボス戦を実装することである。
