# モンスター育成RPG：iPhone向けPWA実現可能性検討書

作成日：2026年7月12日  
前提：販売なし／個人利用／Windowsで開発／iPhoneでプレイ

## 結論

**個人用・1人プレイ・2D・ターン制の範囲なら、MacがなくてもPWAとして十分実現可能。**

ただし、「ドラゴンクエストモンスターズの複製」ではなく、次のような**オリジナルのモンスター育成RPG**として制作することを推奨する。

- モンスターを仲間にする
- パーティーを編成してターン制で戦う
- 育成した個体から能力・特性・スキルを次世代へ継承する
- 探索地点とボスを順番に攻略する

作品名、モンスター、画像、音楽、文章、画面、計算式、世界観は独自に設計する。既存作品の固有名詞や公式素材は使わない。

## 1. 判断概要

| 判断項目 | 評価 | 要点 |
|---|---:|---|
| 技術的実現性 | 高い | 2D・1人用・ターン制はWeb技術と相性がよい |
| Macなし開発 | 可能 | Windowsで開発・公開し、iPhone実機で確認できる |
| 完成可能性 | 中〜高 | MVPを絞れば現実的。通信や広いフィールドは後回し |
| 権利リスク | 要対策 | 固有名詞・画像・音楽・データを流用せずオリジナル化する |
| 推奨開始案 | Go | React中心の静的PWAで縦切り版を作る |

Macがないことは中止理由にならない。PWAに限定することで、App Store審査、コード署名、Xcode、Apple Developer Programの年会費を避けられる。

## 2. 最初に作るもの

### MVPに含める

- 縦画面を基本とする1人用の2Dターン制RPG
- 「探索地点を選ぶ → 戦闘 → 仲間化 → 育成 → 継承合成 → 次の地点」の循環
- オリジナルモンスター20〜30体
- スキル10〜15個
- 探索地点8か所程度
- ボス3体程度
- 端末内への自動保存
- JSON形式のバックアップ書き出し・読み込み
- ホーム画面からの独立起動
- 初回読み込み後のオフラインプレイ

### MVPに含めない

- オンライン対戦、ランキング、チャット、フレンド
- ユーザー登録、課金、ガチャ、運営用管理画面
- 広大な自由移動マップ
- リアルタイム戦闘、3D表現
- iCloud同期、プッシュ通知
- App Store公開
- 既存作品のモンスター、名称、画像、BGM、効果音、文章、数値データ

これは単なる機能削減ではない。最も重要な「集める・育てる・継承する・戦う」が面白いかを、少ない工数で検証するための順序付けである。

## 3. iPhone向けPWAの実現性

AppleのWebKitは、ホーム画面へ追加したWebサイトを独立したWebアプリとして起動する仕組みを提供している。iOS 26では、ホーム画面へ追加したサイトは原則としてWebアプリとして開く挙動が案内されている。[WebKit: Safari 26.0の変更点](https://webkit.org/blog/17333/webkit-features-in-safari-26-0/)

### 機能別評価

| 要件 | 可否 | 実装方法 | 注意点 |
|---|---:|---|---|
| ホーム画面起動 | ○ | Web App Manifestとアイコン | 追加操作は利用者が行う |
| オフライン動作 | ○ | Service Workerで本体と素材をキャッシュ | 初回読み込みはオンライン |
| ターン制戦闘 | ○ | HTML/CSSまたはCanvas | 重い3Dは対象外 |
| 端末内セーブ | ○ | IndexedDB | データ消失への備えが必要 |
| BGM・効果音 | ○ | Web Audio / HTML Audio | 最初のユーザー操作後に開始 |
| バックグラウンド進行 | △ | 復帰時に経過時間から再計算 | 常時実行を前提にしない |
| プッシュ通知 | ○ | Web Push | ホーム画面追加と許可操作が必要 |
| 完全に消えない保存 | △ | 永続化要求＋外部バックアップ | ブラウザデータは退避され得る |

### PWA向けの設計原則

**バックグラウンドで動き続けるゲームではなく、いつ中断されても同じ状態から再開できるゲームにする。**

具体的には次の対策を行う。

1. 戦闘開始、ターン確定、勝敗確定、仲間化、合成の境界で自動保存する。
2. セーブに `schemaVersion`、`appVersion`、`savedAt` を持たせる。
3. 起動時に古いセーブデータを新しい形式へ移行する。
4. `navigator.storage.persist()` で永続化を要求する。
5. `StorageManager.estimate()` で使用量と割当量を診断できるようにする。
6. JSONバックアップの書き出しと復元を設定画面に常設する。
7. `visibilitychange` と `pageshow` を監視し、復帰時に画面と音声を再同期する。
8. 新しいService Workerを戦闘中に即時適用しない。

WebKitは、IndexedDB、Cache API、Service Worker等のデータが、容量不足やストレージ圧迫等によって退避される可能性を説明している。そのため、端末内セーブだけを唯一の正本にしない。[WebKit: Storage Policy](https://webkit.org/blog/14403/updates-to-storage-policy/)

iOSではバックグラウンド移行時にWeb Audioが停止する挙動があるため、ゲーム復帰時にAudioContextを再開・再構築できるようにする。[WebKit Bugzilla 220879](https://bugs.webkit.org/show_bug.cgi?id=220879)

## 4. 権利面

### 基本方針

販売せず自分だけで遊ぶ場合でも、「非営利なら既存作品を自由に使える」とは限らない。

日本の著作権法には私的使用のための複製に関する例外がある一方、インターネットへのアップロードは公衆送信・送信可能化に関係する。文化庁も、ネット公開する二次創作について、複製権・翻案権等の許諾が原則として必要と説明している。

- [e-Gov法令検索：著作権法](https://laws.e-gov.go.jp/law/345AC0000000048)
- [文化庁：ここが知りたい著作権](https://www.bunka.go.jp/seisaku/chosakuken/taisetsu/point/index.html)

PWAは通常HTTPSサーバーへ配置する。URLを広く告知しない場合でも、他人の画像や音楽をサーバーへ置く設計は避ける。認証を付ければ問題がなくなる、という前提にもしない。

### オリジナル化の基準

| 要素 | 推奨 |
|---|---|
| 作品タイトル | 公式作品や系列作品と誤認されない完全な別名 |
| モンスター名・外見 | すべてオリジナル |
| 画像・アニメーション | 自作または利用許諾が明確な素材 |
| BGM・効果音 | 自作、CC0、またはゲーム利用が許可された素材 |
| システム | 抽象的な着想を基に独自のルールと計算式を作る |
| 説明文 | 自分の言葉で新規作成する |

### 使用しないもの

- 「ドラゴンクエスト」「ドラゴンクエストモンスターズ」の名称
- 公式ロゴ
- 既存モンスターの名称・画像・デザイン
- ゲーム画面から切り出した画像
- トレースした画像
- 公式BGM、耳コピーした楽曲、抽出した効果音
- 図鑑説明やスキル説明の転記
- 公式の配合表、能力値、計算式の丸写し

ゲームの抽象的な楽しさを研究し、名称、世界観、ルール、画面、文章、データを独自に設計する。

> この整理は企画上の一般的なリスク評価であり、個別案件に対する法律相談ではない。

## 5. 推奨するMVP仕様

### コアループ

```text
拠点・編成
   ↓
探索地点を選択
   ↓
1〜3戦のターン制バトル
   ↓
仲間化判定・成長・報酬
   ↓
2体から新個体を誕生させ、特徴を継承
   ↓
次の探索地点を解放
```

1周を10〜15分程度にする。自由移動マップは後から追加し、最初は地点選択式でゲーム性を検証する。

### 最小コンテンツ量

- モンスター24体
  - 4系統 × 6体
  - 基本能力、特性1個、習得候補2〜3個
- スキル12個
  - 攻撃4、回復2、補助4、状態異常2
- 探索地点8か所
  - 通常敵、希少敵、報酬、解放条件をデータ化
- ボス3体
  - 攻撃予告や状態変化で戦術性を持たせる
- 継承合成
  - 親2体から系統、能力補正、特性候補、スキル候補を決定
  - 実行前に結果予告を表示する
  - 理不尽な能力消失を避ける

### MVPの完成条件

- [ ] 新規開始から最初のボス撃破まで15〜30分遊べる
- [ ] アプリを閉じ、機内モードで再起動しても続きから遊べる
- [ ] 戦闘中断でアイテム増殖や進行不能が起きない
- [ ] バックアップを書き出し、データ削除後に復元できる
- [ ] iPhoneの小さい画面でもタップ領域が十分に大きい
- [ ] 既存作品の素材・名称・文章が含まれていない

## 6. 推奨技術構成

### MVP段階

| 層 | 推奨 | 役割 |
|---|---|---|
| 画面 | React + TypeScript | 編成、戦闘、図鑑、合成、設定 |
| ビルド | Vite | 開発サーバーと静的ファイル生成 |
| ゲームロジック | 純粋なTypeScript | 戦闘、仲間化、成長、合成の計算 |
| マスターデータ | JSON / TypeScript | モンスター、技、敵編成、報酬 |
| セーブ | IndexedDB | セーブ、設定、移行情報 |
| オフライン | Service Worker | 本体、画像、音声のキャッシュ |
| テスト | Vitest + Playwright | 計算の単体テストと画面回帰 |
| 公開 | 静的HTTPSホスティング | サーバーロジックなしで配信 |

### Phaser等のゲームエンジンについて

MVPではReactとHTML/CSSを中心にする。メニュー主体のターン制RPGに、最初からゲームエンジンを追加すると、画面遷移・状態・タッチ入力を二重管理することになる。

自由移動するタイルマップが本当に必要になった時点で、Phaser等を探索画面に限定して追加する。

### 責務分離

```text
画面コンポーネント
    ↓ コマンド
ゲームロジック
    ↓ 次の状態＋イベント履歴
セーブ層
    ↓
IndexedDB
```

Service Workerは実行コードと素材のオフライン供給だけを担当する。セーブ本体はキャッシュへ入れない。

### データ設計で先に決めること

- 乱数シードと戦闘再開時の扱い
- セーブデータの版番号
- 変更されない内部ID
- 戦闘計算の再現性
- ゲーム更新中のセーブ移行方法
- Service Worker更新を適用するタイミング

## 7. Macなしでの開発とテスト

Windowsだけで次の作業を行える。

- ソースコード作成
- ローカル開発
- 自動テスト
- PWAのビルド
- HTTPSホスティングへの公開
- Gitによるバージョン管理

PWAのホーム画面利用にXcodeやApple Developer Programへの加入は不要。Appleは、ホーム画面WebアプリのWeb PushについてもDeveloper Program加入不要と案内している。[WebKit: Web Push for iOS and iPadOS](https://webkit.org/blog/13878/web-push-for-web-apps-on-ios-and-ipados/)

### 推奨する確認サイクル

1. WindowsのChromeまたはEdgeでゲームロジックと画面を確認する。
2. 変更をプレビュー用HTTPS URLへ公開する。
3. iPhoneのSafariで開き、ホーム画面へ追加する。
4. ホーム画面版で音声、復帰、オフライン、セーブを確認する。
5. 問題があれば画面内ログまたはエラー書き出しで情報を回収する。

Appleの正式なiOS Web Inspectorは、Mac側Safariの開発メニューからiPhoneを選択する手順である。Macがない環境では、開発版に次の機能を用意するとよい。[Apple Developer: Inspecting iOS and iPadOS](https://developer.apple.com/documentation/safari-developer-tools/inspecting-ios)

- 画面内デバッグログ
- エラー内容のコピー・JSON書き出し
- iOS、画面サイズ、PWA起動状態の表示
- IndexedDBとキャッシュの診断画面
- 音声コンテキストの状態表示

### iPhone実機テスト項目

- [ ] 初回アクセスからホーム画面追加まで
- [ ] ホーム画面からの独立起動
- [ ] 初回タップ後のBGM開始
- [ ] 消音設定と音量設定
- [ ] アプリ切り替え後の音声復帰
- [ ] 機内モードでの起動
- [ ] キャッシュ更新直後の再起動
- [ ] 戦闘中断と端末ロックからの復帰
- [ ] セーブ、バックアップ、復元、旧版移行
- [ ] ノッチと `safe-area-inset`
- [ ] 文字拡大と小さい画面
- [ ] ダークモードと低電力モード

## 8. 工数目安

以下はWeb開発経験者が、簡素なオリジナル2D素材を使う場合の概算。未経験者は1.5〜2倍程度を見込む。

| 段階 | 主な成果 | 累計目安 | 出口条件 |
|---|---|---:|---|
| 0. 企画固定 | 名称、コアループ、権利面の赤線、データ項目 | 4〜8時間 | 1ページ仕様が決まる |
| 1. 技術スパイク | PWA起動、1戦、IndexedDB保存、音声 | 20〜40時間 | 機内モードで再戦できる |
| 2. 縦切り版 | 仲間化、編成、成長、合成、1ボス | 80〜140時間 | 15〜30分の循環が面白い |
| 3. 個人用MVP | 24体、8地点、3ボス、バックアップ | 180〜320時間 | 最初から最後まで遊べる |
| 4. 拡張 | 探索マップ、演出、クラウドセーブ等 | 400時間〜 | 必要性を再評価して選択 |

週6〜8時間なら、縦切り版まで約3〜5か月、個人用MVPまで約6〜12か月が現実的な範囲となる。

## 10. 主なリスク

### スコープ膨張

最大のリスクは技術ではなく、作りたい要素が増え続けること。「最初のボスまで」を独立した完成品として扱い、追加要素を別マイルストーンにする。

### セーブ消失

永続化要求、自動保存、スキーマ移行、手動バックアップを組み合わせる。クラウド同期はMVP後でよい。

### iOS固有不具合

Windowsブラウザだけで完成判定をしない。少なくとも週1回、iPhoneのホーム画面版で音声、復帰、オフライン、セーブを確認する。

### 権利侵害・公開事故

公式作品から抽出・転記した素材を、ローカルの試作、Gitリポジトリ、公開URLのいずれにも入れない。最初からオリジナルまたは明示的に許諾された素材だけを使う。

### バランス調整の迷走

戦闘イベントログと簡易シミュレーターを用意し、勝率、平均ターン数、使用スキル比率を確認する。感覚だけで能力値を変更しない。

## 11. 最終提案

**Go。ただし次の条件を最初の完成線とする。**

- オリジナルIP
- 1人プレイ
- 2D・ターン制
- 地点選択式の探索
- モンスター24体程度
- 端末内セーブ
- JSONバックアップ
- iPhone実機でのオフライン確認

### 次に決める5項目

1. 仮タイトルと世界観
2. 戦闘人数（1対1、3対3など）
3. 仲間化方式
4. 継承合成で親を失うか、何を継ぐか
5. グラフィック方針

この5項目が決まったら、6体・4スキル・1戦・1セーブだけの**技術スパイク**を作る。iPhoneのホーム画面版で動くことを確認した後、正式なゲーム仕様書と実装タスクへ進む。
