# 『封域異獣録』状態管理設計

版：0.1  
更新日：2026年7月13日  
対象：縦切り版

## 1. 採用方針

アプリ起動中のゲーム状態は、React標準の `useReducer` と `Context` でメモリ上に保持する。Dexieは永続保存に限定し、画面表示のたびにIndexedDBを直接参照しない。

```text
React画面
  ↓ コマンド
GameSession Provider
  ↓
純粋TypeScriptのゲームルール
  ↓ 新しい状態＋イベント
useReducerで画面状態を更新
  ↓ 確定保存要求
SaveCoordinator
  ↓
Dexie／IndexedDB
```

この構成では、ゲームの正しい状態を決める場所を次のように分ける。

| 状態 | 正とする場所 |
|---|---|
| 起動中の最新ゲーム状態 | `GameSession` のメモリ状態 |
| アプリ終了後にも残す確定状態 | Dexie／IndexedDB |
| 戦闘や継承の計算規則 | `domain` の純粋TypeScript |
| 一時的な開閉・選択表示 | 各画面のローカル状態 |

## 2. 状態の分類

### 2.1 永続ゲーム状態

セーブ対象となる状態。`GameSessionState` の中心を構成する。

- 主人公設定と物語フラグ
- 所有異獣、編成、育成状態、信頼度
- 異獣録の調査段階
- 研究資源、触媒、所持品
- 地域と地点の進行
- 確定済みの調査状態
- 再開可能な戦闘状態
- 共鳴休止回数
- セーブ形式の版と状態の世代番号

### 2.2 セッション状態

再開に必要なものだけを保存する起動中の状態。

- 現在の大画面
- 調査中の地域と地点
- 戦闘のラウンド、前衛・控え、活性値、状態変化
- ボス戦の段階、過負荷、警戒度
- 最後に確定した操作

画面のアニメーション途中や、押下中のボタン状態などは含めない。

### 2.3 UI一時状態

保存せず、画面または小さなUI用Contextに置く。

- ダイアログの開閉
- タブの一時選択
- 技能説明の表示
- 入力途中の3体分の行動案
- トースト通知
- アニメーション進行度

戦闘の行動案は「ラウンド確定」までゲーム状態へ反映しない。アプリが閉じられた場合は前ラウンド解決後から再開し、未確定の入力は破棄してよい。

## 3. Contextの分割

全画面を1個の巨大なContextへ接続しない。最低限、次に分ける。

| Context | 内容 |
|---|---|
| `GameSessionContext` | ゲーム状態の読取、ゲームコマンドの実行 |
| `AppStatusContext` | 起動、読込、保存中、オフライン、更新待ち、致命的エラー |
| `UiContext` | 全体ダイアログ、通知、音声設定の反映 |

編集中のフォームや画面内タブはContextへ入れず、各画面のローカル状態にする。ゲーム状態も、画面ごとに必要な値へ絞った専用Hookから読む。

```ts
const party = useParty();
const battle = useActiveBattle();
const saveStatus = useSaveStatus();
```

状態全体を任意のコンポーネントから直接書き換えるHookは公開しない。

## 4. GameSessionの基本形

概念上の最小構造は次のとおり。

```ts
type GameSessionState = {
  schemaVersion: number;
  revision: number;
  profile: PlayerProfile;
  creatures: Record<CreatureId, Creature>;
  party: PartyState;
  bestiary: BestiaryState;
  exploration: ExplorationState;
  battle: BattleState | null;
  inventory: InventoryState;
  story: StoryState;
};

type SessionAction =
  | { type: 'hydrate'; snapshot: GameSessionState }
  | { type: 'acceptCommand'; result: CommandResult }
  | { type: 'replaceAfterMigration'; snapshot: GameSessionState };
```

画面は低水準の `SessionAction` を直接送らず、意味のあるゲームコマンドを実行する。

```ts
type GameCommand =
  | { type: 'confirmBattleRound'; actions: PlannedAction[] }
  | { type: 'chooseExplorationAction'; actionId: string }
  | { type: 'requestCooperation'; targetId: CreatureId }
  | { type: 'confirmResonance'; plan: ResonancePlan };
```

ゲームルール層は現在状態とコマンドを受け、検証済みの新状態と表示用イベントを返す。不正なコマンドでは状態を変更せず、画面へ理由を返す。

## 5. 保存の流れ

### 5.1 世代番号

確定した状態変更ごとに `revision` を1増やす。保存済みの世代より古い結果で上書きしない。

```text
revision 20 を保存開始
revision 21 が発生
revision 20 の保存完了
revision 21 を保存
→ 最終保存は必ず21
```

### 5.2 SaveCoordinator

保存要求は `SaveCoordinator` へ集約し、IndexedDBへの書込みを直列化する。

- 同時に複数の保存処理を走らせない
- 待機中に新しい状態が来た場合、不要な中間保存はまとめられる
- 保存対象の `revision` を記録する
- 古い世代を新しい世代の上へ書かない
- 失敗時はメモリ上の最新状態を保持し、再試行できる
- 保存完了前に「保存済み」と表示しない

戦闘終了、仲間化、共鳴継承、バックアップ復元は重要保存として扱い、中間保存へまとめず完了を待ってから次の危険な操作へ進む。

### 5.3 保存状態の表示

`AppStatusContext` は次の状態を持つ。

| 状態 | 表示・動作 |
|---|---|
| `saved` | 最新世代まで保存済み |
| `saving` | 保存中。通常操作は継続可能 |
| `retrying` | 一時失敗。自動再試行中 |
| `failed` | 保存できていないことを明示し、再試行とJSON退避を案内 |

保存失敗を黙って成功扱いにしない。ゲーム進行を即座に破棄もしない。

## 6. 起動と復帰

起動状態は次の順で遷移する。

```text
booting
  → IndexedDBを開く
  → セーブ形式を検査
  → 必要なら移行
  → GameSessionへhydrate
  → ready
```

読込が完了するまでゲーム画面を操作可能にしない。移行失敗時は新規ゲームで上書きせず、エラー内容、JSON書出し、再試行を提示する。

ホーム画面へ戻っただけではセッション状態を破棄しない。ページ再読込またはプロセス終了後は、最後に確定保存された状態から復帰する。

## 7. コマンドごとの確定境界

| 操作 | メモリ更新 | 永続保存 |
|---|---|---|
| 戦闘の行動を1体分選ぶ | UI一時状態のみ | しない |
| 戦闘ラウンド確定 | 解決後の戦闘状態へ更新 | する |
| 技能説明を開く | UI一時状態のみ | しない |
| 地点の選択肢を確定 | 調査結果へ更新 | する |
| 観察演出を見る | 表示のみ | しない |
| 仲間化成立 | 所有個体と図鑑を更新 | 重要保存 |
| 編成を並べ替えている途中 | 編集用状態のみ | しない |
| 編成確定 | 編成を更新 | する |
| 共鳴内容を試算 | プレビューのみ | しない |
| 共鳴継承確定 | 幼体、素材、休止状態を一括更新 | 重要保存 |
| 音量変更 | 設定を更新 | する |

## 8. Dexieの扱い

Dexieは次に限定して使用する。

- 起動時の読込み
- 確定状態のトランザクション保存
- 設定保存
- JSONバックアップ用データの読出し
- セーブ形式の移行

Dexieの `useLiveQuery` は、セーブ一覧や保存メタ情報などDB自体の変化を表示する小さな用途なら使用できる。進行中の戦闘状態や調査状態を `useLiveQuery` から直接駆動しない。

## 9. エラー時の原則

- ゲームルールの検証失敗では状態を変更しない
- 保存失敗でもメモリ上の最新状態を直ちに捨てない
- 部分保存を避け、複数テーブル更新はトランザクション化する
- 復元・移行の失敗で既存セーブを上書きしない
- 予期しないエラーでは、最後の確定保存から安全に再開できるようにする
- 開発版ではコマンド、乱数シード、開始世代を診断情報として残す

## 10. テスト項目

- 同じ状態とコマンドから同じ次状態が得られる
- 不正なコマンドで `revision` が増えない
- 高速に連続保存しても最終世代が残る
- 保存途中で新しい保存要求が来ても順序が逆転しない
- 戦闘の未確定行動がセーブへ混入しない
- ラウンド解決後の再読込で二重報酬が発生しない
- 共鳴継承の個体追加、素材消費、休止設定が一括成功または一括失敗する
- 保存失敗後に再試行できる
- 旧セーブ移行後に同じゲーム進行を継続できる

## 11. 今回採用しない方式

### Zustandなどの追加状態管理ライブラリ

縦切り版の規模ではReact標準機能で管理可能と判断する。Contextの再描画や記述量が実測上の問題になった場合に導入を再検討する。

### Dexieを画面状態の中心にする方式

保存とゲーム進行が密結合になり、戦闘計算のテスト、中間操作、保存失敗時の扱いが複雑になるため採用しない。

### 各画面が個別に保存する方式

保存順序やトランザクション境界が分散するため採用しない。永続化は `SaveCoordinator` を唯一の入口とする。
